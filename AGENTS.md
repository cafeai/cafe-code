# AGENTS.md

## Task Completion Requirements

- When code changes are made, all of `bun fmt`, `bun lint`, and `bun typecheck` must pass before considering tasks completed.
- NEVER run `bun test`. Always use `bun run test` (runs Vitest).
- When a change touches Electron packaging, backend bootstrap, provider daemon startup, or generated bundle boundaries, also run `bun run build:desktop`.
- Rust takes a while to compile. Do not prematurely kill Rust/cargo builds.
- Integration tests that require live provider binaries, real provider credentials, process reaping, or detached daemon handoff should be explicit `*.e2e.test.ts` or documented opt-in commands. Do not silently put flaky external-provider assumptions on the default test path.

## Search Discipline

- Prefer `rg` and `rg --files` for source discovery. Do not run broad `find` searches from `/`, `/usr`, `/opt`, `/Users`, or other high-level roots unless the task absolutely requires it and the reason is stated first.
- Keep filesystem searches targeted to the repository subtree or a specific known parent directory. Exclude generated and dependency-heavy directories such as `node_modules`, `target`, `__pycache__`, `.git`, `.turbo`, `.astro`, `dist`, and `build`.
- When `find` is necessary, use bounded roots plus pruning/exclusion predicates instead of scanning the whole machine.

## Windows-Specific Notes

- All Windows-specific requirements, implementation notes, workarounds, and verification guidance must live in this section. Do not scatter Windows-only rules through unrelated architecture, provider, packaging, or lifecycle sections; add or update bullets here instead.
- When making changes on Windows, be overly aware of three priorities: record every Windows-specific behavior or workaround in this section, never regress macOS/Linux behavior, and treat Windows as lower priority than macOS/Linux while still making the Windows path correct and maintainable.
- Keep Windows-only fixes documented in this section before implementation. Scope them so they do not change macOS or Linux behavior unless the task explicitly asks for a cross-platform change. Document how macOS/Linux behavior is preserved, and verify at least `bun run build:desktop` plus the relevant Windows command before considering a Windows fix complete.
- Windows NSIS packaging uses `electron-builder`, which rejects a staged app if `electron` appears in `dependencies`; Electron must be present only in `devDependencies` for the staged builder package. Because `apps/server/package.json` currently lists `electron` among runtime dependencies, `scripts/build-desktop-artifact.ts` must omit `electron` from staged runtime dependencies while still adding the desktop Electron version to staged `devDependencies`. This is a staged package shape fix, not a runtime dependency or platform-target change. It preserves macOS/Linux behavior by leaving shared build config, platform target selection, icons, signing/publish configuration, and non-Electron staged runtime dependencies unchanged.
- Windows Codex shadow homes must not require Developer Mode or administrator symlink privileges. Keep macOS/Linux on the existing symlink-based layout, but on Windows use Windows-safe links for shared entries: directory junctions for shared directories, hard links or copies for shared files, and real copied private files for `auth.json`. Verify the Windows path without changing POSIX shadow-home behavior.
- Windows Store/MSIX Codex installs expose private app resources under `C:\Program Files\WindowsApps` that can appear on `PATH` but fail with Access Denied outside the package identity. Windows-only Codex CLI autodetection should prefer user-owned Codex cache bins under `%LOCALAPPDATA%\OpenAI\Codex\bin\<hash>` ahead of WindowsApps resources. Do not apply this path hydration on macOS or Linux, and keep explicit user-configured `binaryPath` values authoritative.
- Windows provider health checks can receive localized CLI failure text in the active Windows code page instead of UTF-8. Preserve UTF-8 decoding on macOS/Linux, but on Windows fall back only when the byte stream is invalid UTF-8 so settings diagnostics do not show mojibake.
- Windows tests must not assume Developer Mode or administrator symlink privileges. If a test specifically validates symlink handling and Windows returns a symlink privilege `EPERM`, skip only that privilege-dependent assertion path while preserving the full symlink security test on macOS/Linux and on Windows machines where symlinks are available.
- Windows file URL conversion must use `fileURLToPath` instead of `URL.pathname`, because raw pathname strings keep a leading slash before the drive letter. This is safe for macOS/Linux and prevents Windows-only false negatives when tests resolve repo-relative assets.
- Windows test assertions that run through the host `Path.Path` service must build expected paths through that same path service instead of hard-coded POSIX separators. Keep macOS/Linux behavior unchanged by deriving expected strings from the platform path implementation already under test.
- Windows subprocess and heavy dynamic-import tests can need longer per-test timeouts under full parallel `turbo` load. Apply longer timeouts only under `process.platform === "win32"` and preserve existing macOS/Linux timeouts.
- Windows source checkouts may start Cafe through `Start-CafeCode.ps1`, which checks whether `openssl.exe`/`openssl` is available as an application on `PATH` before launch. The launcher should preserve default local HTTPS when OpenSSL is present, and set `CAFE_CODE_HTTPS_ENABLED=false` only when OpenSSL is missing because source installs do not always have it available. Desktop backend configuration must also propagate `CAFE_CODE_HTTPS_ENABLED=false` whenever `DesktopServerExposure` resolves no HTTPS port, while keeping macOS/Linux behavior unchanged by deriving the child environment from the shared exposure config rather than from a platform branch.

## Project Snapshot

Cafe Code is a desktop GUI for coding agents such as Codex and Claude. Cursor and OpenCode provider integrations were retired; their historical settings and persisted thread references must decode safely but should be filtered out of runtime registries and provider settings UI rather than surfaced as unavailable first-party providers. The project is still early and inherited code may be incorrect or overfit to older fork behavior. Prefer evidence-backed fixes, strong diagnostics, and maintainable redesigns over small patches that only hide symptoms.

Core priorities:

1. Performance first.
2. Reliability first.
3. Security first.
4. Treat 16+ hour provider prompts as a normal target workload, not an edge case.
5. Keep behavior predictable under load and during failures such as restarts, provider crashes, partial streams, delayed provider wakeups, and reconnects.

If a tradeoff is required, choose correctness, durability, and debuggability over short-term convenience.

## Security Requirements

- Write all code as if it will run in a security-conscious environment where adversaries will constantly try to attack local transports, provider sessions, provider credentials, persisted command ledgers, and debug surfaces.
- Local provider daemon and supervisor transports must be loopback-only or IPC by default, authenticated with high-entropy capability tokens, and must never be exposed on a non-loopback interface without an explicit authenticated design.
- Secrets such as provider daemon tokens, Codex auth, Claude credentials, and app bootstrap credentials must be stored as private files with restrictive permissions where the platform supports them. Never persist secrets in logs, debug JSON, process argv, or user-visible error strings.
- Do not symlink auth files into shadow homes. Copy private auth files when needed, chmod them to `0600`, and reject unsafe symlinked auth material.
- Debug endpoints may expose operational diagnostics but must not expose raw prompts, full model outputs, bearer tokens, cookies, auth JSON, API keys, or unrestricted filesystem paths unless the user explicitly asks for a local forensic dump.
- Any change touching provider process ownership, daemon handoff, provider-home materialization, approval handling, command execution, or debug output needs a quick security audit and tests for the failure mode being addressed.
- Inform the user when security-related behavior changes.

## Maintainability

- Long-term maintainability is a core priority. If you add functionality, first check for shared logic that belongs in a separate module. Duplicate lifecycle or provider-connection logic across files is a code smell.
- Do not write stubs or incomplete code. Finish the behavior, tests, diagnostics, and cleanup needed for the requested change.
- Prefer structured schemas, generated protocol types, durable state machines, and explicit event names over stringly-typed local conventions.
- Add verbose comments for non-obvious code. Lifecycle, provider connection, daemon/supervisor, auth, persistence, and protocol-boundary code should be especially well commented.
- Comments should explain why a state transition, timeout, reconciliation step, security check, or provider protocol workaround exists. Do not leave lifecycle code with unexplained magic constants or unexplained "best effort" branches.
- When implementing Codex, Claude, or other provider protocol behavior, reference official docs and, where available, the provider's current SDK types or open-source implementation. Record important source-derived assumptions in comments near the code that depends on them.
- Keep this file's architecture sections current. Any change that modifies how a provider starts, resumes, streams, retries, falls back, asks for approval, persists session identity, hands off to a daemon/supervisor, or reports diagnostics must update the matching provider section below so future agents inherit the current design instead of stale fork assumptions.
- Periodically re-check upstream provider implementations when pinned CLIs/SDKs change, especially under-development feature flags, default-enabled feature flags, request config keys, model/tool metadata, and lifecycle contracts. Cafe should intentionally track upstream behavior where it is stable, and should keep local compatibility pins or quarantines documented until upstream behavior is verified safe for Cafe's long-running workloads.

## Package Roles

- `apps/desktop`: Electron main process, desktop bootstrap, debug server, backend process manager, provider daemon manager, and local process reaper. This is where app startup decides whether to adopt, spawn, stop, or leave detached runtime processes alive.
- `apps/server`: Node.js backend and provider runtime host. It serves the web assets, exposes HTTP/HTTPS and WebSocket/API surfaces, owns orchestration, persistence, provider adapters, the provider daemon HTTP/RPC server, and provider supervisor support.
- `apps/web`: React/Vite Electron renderer and browser Web UI. It owns session UX, message timelines, scroll behavior, settings UI, and client-side state. It receives orchestration domain events from the backend and must not invent provider lifecycle state that the backend did not emit.
- `packages/contracts`: Shared Effect Schema contracts for provider runtime events, orchestration events, daemon RPC/health, supervisor metadata, WebSocket/API protocol, desktop bootstrap, settings, and model/session types. Keep this package schema-only; do not add runtime business logic.
- `packages/shared`: Shared runtime utilities consumed by server, desktop, and web. Use explicit subpath exports such as `@cafecode/shared/git`; avoid barrel indexes.
- `packages/effect-codex-app-server`: Typed Codex app-server JSON-RPC client/protocol surface. Keep this aligned with the Codex version we are targeting.
- `packages/effect-acp`: ACP protocol support used by compatible providers.

## Browser Web UI and Local HTTPS

- The browser Web UI is a first-class remote surface. Any shared renderer feature that opens editors, terminals, paths, folders, or OS dialogs must be gated on explicit local host capability and should use copy-path/copy-command alternatives in pure browser or reverse-proxy sessions.
- After an authenticated backend connection is available, renderer `ClientSettings` are backend-authoritative and shared across Electron and browser clients. Local desktop/browser client-settings persistence is only a bootstrap/import fallback before backend settings are available.
- The main backend keeps HTTP available for compatibility and may also serve a self-signed HTTPS/WSS sibling listener on a separate port. In desktop builds, local HTTPS is controlled by an explicit desktop settings toggle, separate from the local-only/network-accessible host exposure toggle. Browser-facing pairing/headless output should prefer HTTPS when it is enabled; Electron desktop bootstrap may keep using HTTP loopback to avoid self-signed certificate handling in the desktop window.
- HTTPS public certificate material belongs under the server TLS directory, but HTTPS private keys are secrets and must live under the server secrets directory with restrictive permissions where supported. Do not log private key material, generated key output, or TLS secrets.
- Provider daemons and provider supervisors must not inherit the app HTTPS listener by default. Keep their existing authenticated loopback or IPC transports unless a separate authenticated TLS design is implemented for those runtime layers.

## Renderer Markdown Math

- Chat messages render Markdown through `apps/web/src/components/ChatMarkdown.tsx`. Provider math output is intentionally normalized before Markdown parsing by `apps/web/src/lib/chatMarkdownMath.ts`: Codex commonly emits fenced `math`/`tex`/`latex` blocks, while Claude commonly emits `$...$`, `$$...$$`, `\(...\)`, `\[...\]`, and TeX macros in prose. Preserve all of those shapes when changing message rendering.
- Math rendering uses `remark-math` plus KaTeX through `rehype-katex`; keep KaTeX configured with `trust: false` and non-throwing error handling so malformed or hostile math cannot crash chat rendering or enable trusted HTML-style extensions. Real code fences must remain code fences and must not be reinterpreted as math.
- Native chat selection copy is governed by the local `chatCopyFormat` client setting. The default is Markdown source copy for complete single assistant-message selections, using the same math normalization and provider cleanup as rendering/copy buttons so LaTeX remains pasteable into Markdown renderers. The plain-text setting must leave Ctrl-C and right-click Copy on normal browser visible-text behavior.
- Wide Markdown tables/charts in chat must scroll horizontally inside their own message-local container. Do not let tables, display math, long code identifiers, or long URLs widen the message timeline, crush columns into unreadable letter wrapping, or require page-level horizontal scrolling.
- Table overflow belongs to the table scroll wrapper, not to individual cells. Plain prose cells may stay capped and wrap, but cells containing code spans or KaTeX must be allowed to widen the table so unbroken tokens and rendered formulas do not paint across neighboring columns.
- GFM table cells cannot safely contain true block math. If providers emit same-line display delimiters such as `$$...$$` or `\[...\]` inside a table row, normalize them to inline math before Markdown parsing so the table remains structurally valid after reload.
- Codex ChatGPT search output can include private-use citation delimiters such as `U+E200 cite U+E202 <handle> U+E201` around opaque handles like `turn4search3`. These are valid UTF-8 provider markup, not mojibake or a local decoder failure. Cafe must keep raw provider text persisted for diagnostics, but Codex chat display and assistant-copy text should normalize those markers through `apps/web/src/lib/codexCitations.ts` so private-use glyphs do not appear in the UI. Do not fabricate external links unless app-server supplies trusted URL metadata for the handle.

## Current Runtime Architecture

Cafe Code has three important runtime layers:

- Electron desktop process: starts the UI, starts/adopts the main backend, starts/adopts the provider daemon, exposes a local debug endpoint, and performs process cleanup when the user exits.
- Main backend/server: owns app-level orchestration, event sourcing, projections, settings, WebSocket pushes, HTTP routes, and durable persistence.
- Provider runtime process: a provider daemon, and optionally a persistent provider supervisor, owns long-running provider adapters and live sessions so provider work can survive UI/backend restarts when possible.
- The desktop backend manager treats sustained repeated backend HTTP health failures as a terminal backend run condition even if the child process remains alive. This prevents a split-brain state where the backend PID is still present but the HTTP listener is gone, leaving the renderer unable to reconnect. The post-readiness watchdog is intentionally more patient than bootstrap readiness because large long-running workspaces can briefly push SQLite/projection work over a one-second health probe. Do not lower the watchdog to short one-second failure windows; brief missed probes should log and recover, while a sustained outage should still restart the backend. On replacement startup, stale desktop backend children are reaped before binding the new backend; provider daemon/supervisor processes are intentionally excluded so provider sessions can survive the recovery.

- Projection and renderer backpressure are first-class reliability concerns. Long-lived threads can accumulate hundreds of megabytes of activity payload JSON, so hot-path projections must avoid global message scans, avoid loading full activity payloads for shell summaries, avoid collapsed-row DB fetches from the renderer, and keep cleanup/status work such as remote VCS refresh and checkpoint ref pruning off provider turn completion paths. Message submit, steer, and queue paths must pin the timeline to the conversation tail without trusting stale virtualizer scroll measurements, because a false "not at end" report can trigger expensive historical rendering and DB work when the user scrolls back down. Current mitigations include thread-scoped message lookup for streaming deltas, SQL-side pending user-input accounting, hot-path projection indexes, turn-scoped work-log page indexes, VCS remote refresh single-flight/cooldown, collapsed historical work-log rows that use snapshot lower-bound counts until expanded, submit-time bottom pinning, and asynchronous checkpoint ref pruning.
- Startup and restart recovery must use shell/detail-scoped projection queries. Do not call the full orchestration snapshot from provider reactors, boot recovery, session reconciliation, health checks, or renderer bootstrap paths; on large long-running databases it can hydrate every persisted message and work-log row into the Electron Node heap and crash startup. Use shell snapshots to identify candidate threads, then bounded thread-detail or turn-page queries for the few threads that truly need body/activity data.

## Lifecycle Model

- The renderer requests intent; it does not own provider truth. User actions become orchestration commands/events, then `ProviderCommandReactor` calls `ProviderService`, which routes to the provider daemon or local adapter.
- Provider adapters emit canonical provider runtime events. `ProviderRuntimeIngestion` records them, and projections turn them into renderer-visible state.
- `ProviderRuntimeIngestion` is the owner of ordered provider event flush-through for renderer-visible text. Other provider-event consumers such as `CheckpointReactor` may observe the same `turn.completed` event first, but they must not terminalize a turn from that independent observation until ingestion has published the matching process-local `provider.turn.ingestion-quiesced` receipt. That receipt means Cafe has processed provider events through the terminal event, flushed assistant/proposed-plan buffers, used any authoritative `item.completed` text, and persisted the provider-daemon cursor for that event. If the receipt never arrives, log the lifecycle problem and avoid dispatching terminal checkpoint/diff state from the checkpoint path rather than closing the turn early and truncating text.
- Provider daemon runtime event replay is a bounded live window, not permanent chat history. Durable provider daemon journals must prune to the configured capacity per runtime owner, but that cleanup must run after daemon socket readiness and in bounded batches rather than inside migrations or pre-listen startup. Replay/backfill code must not make later Codex turns pay for every historical token/runtime event Cafe has ever observed. The main backend persists a provider-daemon runtime ingestion cursor after canonical provider events are processed, then reconnects with only a small replay overlap; do not reset remote provider event streaming to cursor `0` on every backend restart. Provider daemon event persistence is idempotent by canonical runtime `eventId`, so supervisor replay after daemon restart fills gaps without appending the same retained supervisor window again.
- Database migrations run before backend and provider-daemon sockets are ready. Keep them to schema changes, indexes, and small bounded data repairs. Do not run large projection rebuilds, JSON event-store scans, DB-wide backfills, or provider-history reconstruction in startup migrations; those repairs must run after readiness as bounded background/manual jobs so desktop health checks and long-lived provider sessions are not blocked.
- Desktop/provider-daemon/provider-supervisor bootstrap data moves over inherited file descriptors, not argv or environment, so capability tokens are not exposed through process listings. On macOS, inherited child-process pipe descriptors can report as sockets; bootstrap readers must consume socket descriptors directly instead of opening `/dev/fd/<n>`, because duplicating those descriptors can block before timeout handling and leave startup waiting forever for an IPC socket.
- Desktop startup hydrates the login-shell environment once before spawning runtime children. Desktop-spawned backend and provider-daemon children receive `CAFE_CODE_SHELL_ENV_HYDRATED=1`, and the backend must skip its own blocking login-shell PATH probe when that marker is present. Direct server CLI launches without the marker still repair PATH themselves.
- Desktop starts the provider daemon eagerly and overlaps provider daemon health readiness with backend startup. The provider daemon manager publishes an authenticated IPC endpoint as soon as the child is spawned and the private credential/marker are written; backend bootstrap must wait for that endpoint before resolving `DesktopBackendConfiguration`, but it does not need to wait for full daemon health. The backend remote provider client is expected to tolerate the daemon listener still coming up and to retry event streaming.
- Desktop source-install update checks run only for local git checkouts on `main` or `dev`. `DesktopSourceUpdates` fetches the matching `origin/<branch>` ref at startup, every 30 minutes, or when the Settings button asks for a manual refresh, then compares `HEAD` with the fetched remote commit while reporting dirty working-tree state separately. Do not auto-pull, merge, stash, or toast from this checker; it is informational UI only. Settings may show the full hash/dirty details, and the chat header may show a passive low-emphasis badge only when the tracked `main`/`dev` branch is behind. Non-`main`/`dev` branches must be reported as ignored without fetching and must never show the chat-header badge.
- Distinguish every lifecycle phase explicitly: user requested, command accepted, provider session connecting, provider session ready, provider turn accepted, provider runtime started, first provider message/token/tool event, completion/interruption/failure, post-completion late event, and reconciliation.
- Do not mark a turn as truly running merely because a send call returned. Some providers ACK a request before their runtime starts streaming. Track ACK and runtime-start as separate facts.
- Terminal UI state must come from provider completion, interruption, failure, or a documented reconciliation path. If a provider sends content after a terminal event, preserve the content and emit diagnostics rather than silently corrupting the turn.
- When a turn becomes terminal, projections must close any still-streaming assistant messages for that turn. Providers can stop through completion, checkpoint finalization, interruption, or failure without sending a separate non-streaming replacement for every partial message; stale `streaming=true` rows must not keep the renderer's working marker alive after provider truth is idle.
- Provider assistant streaming is append-only, but provider `item.completed` payloads can carry authoritative full assistant item text. If the renderer projection has only a streamed prefix, or a streamed prefix plus a small coalesced buffered tail, append the missing completion suffix before the terminal marker; do not let a completed Codex/Claude item leave only the first streamed chunk visible after reload.
- Historical assistant-message repair is explicit and user-triggered from the thread context menu. The renderer may ask the backend to repair every eligible completed assistant message in one selected thread, using retained local `provider_daemon_events` first and then provider-routed upstream history such as Codex app-server `thread/read` when local journal data is missing. It must run after readiness, stay bounded to the selected `threadId`, require terminal non-streaming assistant messages, and apply only prefix-safe suffixes from retained `item.completed` `assistant_message` events or upstream `agentMessage` snapshots. It must not scan all provider history at startup, call raw provider/OpenAI APIs outside the configured provider adapter, repair all threads, reopen terminal turns, restore `activeTurnId`, or mark repaired messages streaming. RPC results, logs, toasts, progress UI, audit activity, and diagnostics may include safe metadata such as counts, lengths, provider, item id, source type, source event id, and result code, but must not expose raw provider text or repair suffixes outside the normal authenticated message update event that applies the repaired content.
- A terminal turn is immutable provider truth for lifecycle purposes. Late or replayed `running` session snapshots for the same provider turn id must not reopen the turn, restore `activeTurnId`, or regress the thread session; a provider-owned new run must arrive under a fresh turn id. If a turn has terminal checkpoint metadata such as `checkpoint_status = ready`, reconcile any reopened `running` row back to terminal before renderer or command-gate state reads it.
- Provider runtime replay/backfill content for a completed turn is reconciliation data. Preserve any snapshot-only messages or activity, but do not reopen the thread session, mark the turn running, or restore `activeTurnId` from post-completion content/tool events.
- Provider runtime event fanout is a critical reliability boundary. The provider adapter to `ProviderService` fanout and the provider daemon/supervisor to persistent journal bridge must treat malformed individual events, logging failures, SQLite hiccups, and stream termination as recoverable. Log and drop only the bad event, restart terminated adapter/journal subscriptions while the same provider instance is still current, and keep later valid provider events flowing. A live Codex/Claude process may keep writing provider logs after this bridge dies; if the journal stalls while provider logs advance, the UI can show phantom running turns even though upstream completed. Bridge diagnostics must stay metadata-only unless the user explicitly asks for a local forensic dump.
- Late streaming deltas or provider replay for a terminal turn must not reopen assistant `streaming` flags. The projection may preserve snapshot-only text when no completed message exists, but completed message rows stay non-streaming and terminal lifecycle remains authoritative, matching the Codex CLI's ordered terminal view.
- Provider runtime replay/backfill can append older logical events after a newer turn has completed. Projection shell fields such as `latest_turn_id`, session status, active turn, and lifecycle `updated_at` must be monotonic by provider turn request time/observed time, not blindly by append order. Stale checkpoint/session backfill is diagnostic/reconciliation data and must not move the renderer back to an older provider-owned turn.
- A newer concrete turn is durable evidence that an older same-thread running turn is stale. Reconcile those non-latest running rows to `interrupted` at the first newer turn request and clear their assistant streaming markers so renderer lifecycle state cannot show phantom provider work.
- Steer/follow-up behavior must respect active-turn invariants. Starting a new turn while one is starting or running is a bug unless it is explicitly queued or converted into a provider-supported steer operation.
- Cafe thread history is provider-agnostic, but live provider sessions and resume cursors are provider-specific. Users must be allowed to switch an idle thread to any configured provider instance; when the target provider driver or continuation identity differs from the existing session, keep the durable thread/messages/work log, start a fresh provider session for the next turn, and never carry a Claude/Codex/custom-instance resume cursor across that boundary.
- Provider resume cursors are optimistic hints, not user-visible failure points. If an upstream provider proves a persisted cursor is invalid for the current provider home/account, such as Codex app-server returning `no rollout found for thread id` after a Codex home/account switch, Cafe must log the reconciliation, retry once with a fresh provider session, and durably replace or clear the stale cursor. Do not surface that recoverable stale-cursor condition as a blocking toast or chat failure.
- Duplicating a thread is a Cafe context fork, not a provider runtime fork. The duplicate may copy persisted messages, terminalized turn/checkpoint context, proposed plans, and historical non-callback work-log rows, but it must not copy provider sessions, provider resume cursors, active turn ids, pending approvals, or pending user-input callbacks. Any copied `running` or `pending` turn state must be terminalized as `interrupted` in the duplicate so the next user prompt starts a fresh provider session from copied context rather than steering or resuming the source provider process.
- Pending approvals and user-input callbacks are provider-runtime state. After restart or daemon handoff, stale callback IDs may be invalid; convert those failures into clear lifecycle events and user-visible recovery guidance.
- Reconciliation after restart must prefer durable provider/session state and command ledgers over UI assumptions. Never automatically resend a user prompt after a crash unless the design cryptographically or durably proves the provider never received it.
- WebSocket shell/detail subscriptions must be replayable from persisted orchestration-event cursors, not hot PubSub-only. A thread detail snapshot's `snapshotSequence` must be read in the same database transaction as the detail payload; never stamp stale detail rows with a newer cursor because the client will correctly skip events at or below that cursor.
- When durable provider runtime state says a session is `stopped`, that runtime record is authoritative over stale projection state. Clear any projected active turn, mark the abandoned projected turn `interrupted` rather than completed, preserve provider resume cursors, and let the next user send resume through a fresh provider process. This follows the Codex CLI/app-server ownership model: a stopped app-server process cannot still own a steerable turn.
- A stale `projection_turns.state = running` row is not sufficient proof that a provider turn is still steerable after restart, interrupt, or stop-all. If durable runtime state is stopped with no active provider turn, startup reconciliation must interrupt pre-stop running turns and close their streaming assistant rows; new user prompts must start/resume fresh provider state instead of being converted into steers to a dead app-server.
- Shutdown/finalizer code must persist `status = stopped` and `activeTurnId = null` before calling provider adapter stop/stopAll methods. If a malformed shutdown leaves `lastRuntimeEvent = provider.stopAll` behind with a stale `running` status, startup reconciliation must treat that marker as authoritative stopped runtime state and repair projections before the renderer can show a phantom active turn.

## Codex Integration

Cafe Code's Codex adapter uses `codex app-server`, which speaks JSON-RPC over stdio by default. Official docs describe the app-server protocol as JSON-RPC-like messages where requests have `id`, responses echo the `id`, and notifications omit `id`. The official app-server websocket listener is experimental/unsupported; Cafe should keep using stdio JSONL for local app-server control unless upstream changes that recommendation.

The generated `packages/effect-codex-app-server` protocol surface is currently aligned to upstream Codex `rust-v0.138.0` commit `c18e9f478bc940ef1ef8e1c426364c0fe3d86b73`. That upstream release preserves the `turn/start`, `turn/steer`, `turn/completed`, and `thread/read` lifecycle contracts used by Cafe, adds client requests such as `account/usage/read` and `skills/extraRoots/set`, and adds notifications such as `turn/moderationMetadata`. Regenerate this package when bumping the local Codex CLI/app-server target rather than hand-editing protocol types.

Cafe Code's Codex UX and lifecycle design should follow the Codex CLI/TUI command-line implementation, not Codex GUI behavior. The CLI is the primary upstream comparison point because it has the operational model Cafe Code needs: user input is accepted into local queues, live `turn/steer` is attempted only against a known active turn, stale active-turn state is reconciled automatically, non-steerable review/compact turns re-queue the user's input, and local app-server/session state is treated as disposable while durable Codex thread identity is preserved through upstream thread state.

Codex lifecycle facts to preserve:

- A Codex connection must send `initialize`, then `initialized`, before other requests.
- `thread/start` creates a new thread and emits `thread/started`.
- `thread/resume` reopens an existing thread so later `turn/start` calls append to it. If the returned thread snapshot contains an `inProgress` turn, Cafe restores `session.activeTurnId` from that turn and keeps polling `thread/read`; if it does not contain an in-progress turn, Cafe must not resurrect stale UI active-turn state.
- `turn/start` accepts input and returns an initial in-progress turn, but that response is a submission ACK, not proof that the new turn has become app-server active truth. The client must keep reading notifications such as `turn/started`, `item/started`, `item/completed`, `item/agentMessage/delta`, `turn/diff/updated`, and `turn/completed`.
- Cafe must mirror the Codex TUI's runtime workspace-root handling. For Codex `thread/start`, `thread/resume`, and new `turn/start` requests, send the current `cwd` plus configured additional project roots as experimental `runtimeWorkspaceRoots` after initializing with `experimentalApi: true`; for workspace-write modes, still send the additional roots as sandbox writable roots. Do not send workspace-root, model, effort, sandbox, or collaboration overrides on `turn/steer`, because upstream TUI steer only injects user input into the active turn.
- If Codex returns a provisional `turn/start` or `turn/steer` ACK turn id and the later app-server `turn/started` notification reports a different provider-owned turn id, treat the app-server notification as authoritative when the runtime session directory agrees. Repair Cafe's projected active turn to the concrete provider turn, retire the provisional active turn as stale, and clear renderer pending-steer markers from the explicit Codex steer-processing activity even if it lands under the repaired turn id. This matches the CLI/TUI rule that request ACKs are not lifecycle truth.
- `turn/steer` appends input to the active in-flight turn, requires the expected active turn id, does not emit a new `turn/started`, and does not accept turn-level overrides.
- Cafe must route live follow-up messages through the separate `ProviderService.steerTurn` / provider-daemon `steerTurn` command path, not through `sendTurn`. The request shape must be only thread id, expected active turn id, user input, and attachments; model, effort, sandbox, collaboration/interaction mode, and other `turn/start` overrides must not be forwarded.
- When provider runtime state says Codex still has a running active turn, that runtime active turn is more authoritative than a stale or newly-starting Cafe projection. Do not call `turn/start` in that state. Convert the user's input to `turn/steer` against the runtime active turn, keep model/effort/mode overrides out of the steer request, and clear the projected pending start only when the steer or recovered `turn/start` path has been accepted. This mirrors upstream TUI `thread_routing.rs`, where submitted user input first tries active-turn steer and only falls through to `turn/start` after app-server reports `no active turn to steer`.
- `ProviderService.sendTurn` is also a recovery boundary, not just a validator. If Codex adapter state still reports a running active turn when a `sendTurn` reaches the provider daemon, route the same user input through adapter `steerTurn` with the reported active turn id instead of throwing `Cannot start a new Codex turn while active turn ... is running`. This handles stale backend/projection/runtime races where the command reactor did not see the daemon's newest active-turn state before issuing `sendTurn`, and matches the Codex CLI/TUI rule that user input is accepted locally and then routed to active-turn steer when upstream still owns a turn.
- The steer split is based on the official Codex app-server docs and the upstream `openai/codex` app-server implementation: `turn_steer_inner` records the request against `expected_turn_id`, calls `thread.steer_input`, returns the active `turnId`, and rejects missing, mismatched, non-steerable, or empty input. Treat the absence of a new `turn/started` after steer as expected protocol behavior, not a stalled start.
- Upstream Codex TUI treats steer failures as active-turn reconciliation: if there is no active turn, it clears its cached active turn and starts a new turn with the same user input; if the active turn is not steerable, it queues rejected input. Cafe Code must not infer steerability from whether its projected assistant message is currently streaming. Official app-server docs bind `turn/steer` to the active in-flight turn plus `expectedTurnId`, and upstream schema/code reserve `activeTurnNotSteerable` for non-regular active turns such as `review` and manual `compact`. Keep those rejection/recovery reasons visible as work-log or debug diagnostics and retry metadata, automatically resend or re-queue the follow-up, and do not expose them as blocking user-facing copy or failure toasts.
- A stale `thread.turn-steer-requested` event can survive a restart or reconciliation boundary after the projected/provider active turn has already been cleared. Treat that preflight condition the same way upstream Codex TUI treats app-server `no active turn to steer`: emit a non-blocking work-log diagnostic and submit the same user message as the next `turn/start`. Do not append a failed-steer activity or require the user to resend merely because the local active-turn cache was repaired before the reactor drained the steer event. This recovery must call the provider send path directly with a short-lived idempotency guard; do not re-dispatch the same message through `thread.turn.start`, because a stale projected active turn can make the decider convert it back into another steer and create an infinite reporting/recovery loop.
- On startup, reconcile the persisted post-terminal stale-steer shape left by older builds: if a thread's latest turn is terminal, no provider runtime still owns it, a later user message is attached to that terminal turn, and Cafe already recorded the stale-steer recovery diagnostic for that message, clear the stale active-turn pointer and dispatch one deterministic `thread.turn.start` command for that same message. This is a bounded repair for a proven local routing bug, not a general crash-resend policy; do not resend arbitrary prompts unless durable state proves Cafe, not the provider, stranded them.
- A successful `turn/steer` response means Codex accepted input for the active turn; it is not proof that the appended user message has reached the provider item stream. Cafe tracks each accepted steer until the matching Codex `userMessage` `item/started` notification appears, emits `codex.turnSteer/noProviderItemYet` warnings after bounded delays, then emits `codex.turnSteer/processingStarted` with ACK-to-provider-item latency. These diagnostics must never include raw prompt text.
- Upstream Codex `rust-v0.138.0` stores accepted steer input in the active turn queue and only drains that queue from the turn loop after current model/tool work reaches a drain point. An ACKed steer with no immediate `userMessage` item is therefore not automatically lost. When delayed steer or snapshot polling sees live app-server descendant processes, Cafe must separate active turn work from persistent Codex helper/MCP/support processes such as bundled `node_repl`, `SkyComputerUseClient mcp`, nested `codex app-server --listen stdio://`, or the npm-wrapper child `codex app-server` runtime; only active child processes justify saying the steer is queued behind child work. Do not auto-interrupt a provider-active turn solely because the user-message item has not appeared yet.
- If a Codex turn becomes terminal before an accepted steer emits its provider-side `userMessage`/processing marker, Cafe must preserve the user's input and requeue the pending steer as the next follow-up instead of clearing the steering marker. A terminal turn alone is not proof that Codex consumed an ACKed steer.
- Upstream Codex TUI treats ACKed-but-uncommitted steers as pending local input. If the user interrupts while pending steers exist, it arms `submit_pending_steers_after_interrupt`; only after the provider interrupt has completed and Codex has cleared pending input may Cafe merge those steers into one next user turn and submit it ahead of ordinary queued follow-ups. Never blind-resend an ACKed steer before that interrupt boundary.
- Automatic Codex compaction is observed as a `contextCompaction` `item/started` to `item/completed` lifecycle on the active turn. With Codex CLI/app-server `0.133.0`, sending `turn/steer` while that item is active can be ACKed by app-server and then leave the turn in progress without a useful follow-up stream or `turn/completed`. Cafe therefore treats an active `contextCompaction` item as the same temporary `activeTurnNotSteerable compact` precondition before sending `turn/steer`, emits `codex.turnSteer/deferredDuringContextCompaction`, and lets the queued-follow-up path resend after the active turn finishes.
- Upstream Codex `0.133.0` only auto-compacts when resolved model metadata or request config provides `model_auto_compact_token_limit`; current `gpt-5.5` metadata can report a large context window while leaving that limit null. Cafe supplies `model_auto_compact_token_limit = 200000` and `model_auto_compact_token_limit_scope = "total"` through `thread/start` and `thread/resume` request config so Cafe-managed sessions compact automatically without editing the user's shared `~/.codex/config.toml`. This threshold follows OpenAI's current public compaction guide example for long-running Codex-style Responses loops plus upstream Codex compact tests, and leaves headroom under the app-server-reported 272k input / 258.4k effective `gpt-5.5` window. A GPT-5.3-Codex system-card evaluation mentioned 100k compaction in an external benchmark harness, but the pinned CLI/app-server source does not expose that as the `gpt-5.5` default.
- Cafe-managed Codex threads explicitly pass `features.remote_compaction_v2 = false` in `thread/start` and `thread/resume` request config. Upstream Codex `rust-v0.136.0` still marks `remote_compaction_v2` as under-development and disabled by default; the v2 remote compaction path streams through the normal Responses tool plan, so it can inherit the stable hosted image-generation tool that currently targets `gpt-image-2` and fail text compaction for accounts/models without that image model. Keep Cafe on upstream's legacy `/responses/compact` behavior until v2 is stable and its tool isolation is verified.
- On every Codex CLI/app-server target bump, re-audit quarantined upstream features before removing compatibility pins. For `remote_compaction_v2`, verify the upstream feature stage/default, whether v2 compaction still inherits normal turn tools, whether hosted image generation still targets `gpt-image-2`, and whether a live long-context compaction smoke test completes without image-tool errors while preserving steer/follow-up behavior.
- Context compaction must be visible to the renderer as work/tool activity. Codex `contextCompaction` items normalize to canonical `context_compaction`; ingestion maps their started/completed lifecycle into `tool.started`/`tool.completed` activities so the chat tool-call/work-log section can show `Context compaction started` and `Context compacted`.
- `turn/interrupt` must include the exact active provider turn id. Upstream Codex rejects missing or mismatched turn ids, and an interrupted turn is not terminal for Cafe until Codex emits `turn/completed` or `thread/read` reports a terminal status for that turn. If Cafe projection and provider runtime disagree, interrupt the provider-runtime active turn first and retry once on the app-server-reported mismatch turn id; app-server ownership is authoritative for interrupt targeting.
- After a successful Codex `turn/interrupt`, retire Cafe's local app-server process and make the next user intent resume through a fresh app-server. This mirrors the CLI/TUI bias toward simple disposable local runtime state and avoids reusing a process that can still ACK later `turn/start` calls while no longer delivering provider item streams.
- Approvals are server-initiated JSON-RPC requests and must be answered through the matching request id.

Codex transport behavior to preserve:

- There are two separate transports. App-server control is local JSON-RPC over stdio. Model streaming inside Codex may use the Responses WebSocket transport when the resolved Codex model provider has `supports_websockets = true`.
- Upstream Codex decides whether Responses WebSockets are enabled with provider capability plus a session-scoped fallback flag. Once `try_switch_fallback_transport` activates, later model requests in that Codex session use HTTP Responses.
- Upstream Codex does not treat `Reconnecting... N/5` as fallback. It retries according to the provider stream retry budget, then emits a warning like `Falling back from WebSockets to HTTPS transport...` when fallback actually activates.
- Cafe must match upstream Codex TUI by leaving official fallback scoped to the live app-server session by default. Do not persist WebSocket fallback across Cafe/app-server restarts or launch future sessions with a Cafe-scoped HTTP-only provider merely because an older session emitted a fallback warning. A legacy diagnostic opt-in, `CAFE_CODE_PERSIST_CODEX_HTTP_FALLBACK=1`, may still persist the official fallback warning and force future launches onto HTTPS for local debugging only.
- When the diagnostic fallback persistence opt-in is enabled and Cafe observes official fallback inside an app-server session that was launched before the persisted policy existed, preserve the active turn, then retire that app-server after a terminal turn event so the next turn resumes through a fresh process with Responses WebSockets disabled. Do not use that opt-in behavior as the normal architecture.
- Codex built-in provider IDs are reserved and cannot be overridden directly. When the explicit diagnostic opt-in needs to start a future app-server with WebSockets disabled, use a Cafe-scoped OpenAI-compatible provider that preserves OpenAI/ChatGPT auth and the Responses API while setting only `supports_websockets = false`.
- Raw Codex `process/stderr` WebSocket connect/stream messages are duplicate tracing diagnostics, not terminal provider failure. Drop the known `codex_api::endpoint::responses_websocket` connect-failure stderr line, keep structured Codex `error` notifications with `willRetry: true` as retry work-log/status diagnostics, keep the official fallback `warning` visible, and rely on `willRetry: false`, process exit, or terminal turn events before surfacing `runtime.error`.
- Upstream Codex `rust-v0.138.0` marks `turn/moderationMetadata` experimental and the TUI routes it to the thread without rendering a chat/work-log cell. Cafe should decode the notification for protocol compatibility but drop it before runtime activity persistence so arbitrary moderation metadata is not written to debug/work-log payloads.
- Keep Codex turn diagnostics explicit: `turn/start` ACK latency, `turn/started`, accepted `turn/steer` versus first steer `userMessage` item, first assistant delta/item, assistant delta cadence/chunk sizes, retry warnings, fallback warning, stream disconnects, and `turn/completed` are separate facts and should be visible in debug output.
- Codex app-server emits fine-grained `item/agentMessage/delta` notifications for live clients. Cafe Code must show the first assistant bytes promptly, but it must coalesce later tiny assistant-text deltas before they cross the durable orchestration/projection boundary. Do not write `thread.session-set` heartbeats or full accumulated message rows for every token; upstream Codex CLI consumes those deltas directly instead of persisting per-token lifecycle state, and Cafe's SQLite/projection path must avoid quadratic rewrite behavior on long answers.
- Upstream Codex CLI/TUI treats accepted steers as pending input until app-server emits the committed `userMessage` item. Cafe should mirror that: keep the steer visible as pending/steering, mark it committed only on the provider item notification, retry once when app-server reports a stale active-turn mismatch, and queue/retry non-steerable review or compact turns without telling the user their message cannot be sent.
- Codex snapshot backfill is reconciliation data, not a second authoritative assistant stream. If delayed snapshot `agentMessage` items repeat assistant text that the live stream already projected for the same turn, keep the live `msg_*` message and suppress the snapshot `item-*` duplicate while still preserving snapshot-only messages.
- Codex session reaping must match the upstream TUI/app-server unload shape: a Codex thread is eligible to stop only after it has no thread-detail subscribers and no provider/thread activity for the unload window, with the effective target computed as `max(noSubscribersSince, inactiveSince) + delay`. Cafe tracks backend thread-detail WebSocket subscribers for this decision, uses provider-session `lastSeenAt` as the inactive clock, and continues to skip any thread with an active turn. Do not apply this subscriber-aware gate to Claude unless its upstream lifecycle requires it; non-Codex providers stay on the standard provider-session reaper.
- Codex `thread/read` snapshots must mirror upstream 0.133.0 stale-turn handling: if a turn snapshot still says `inProgress` but the thread status is no longer `active`, treat that turn as interrupted and emit the terminal snapshot event before clearing Cafe's active-turn state. In-progress snapshots with `active` thread status are not terminal; keep waiting and emit work-log/debug diagnostics that include thread status and item summary.
- Codex delayed snapshot/watchdog polling uses absolute elapsed deadlines from the original turn/steer ACK, not cumulative sleeps. This matches the CLI/app-server model: keep waiting for upstream `turn/completed`, but poll `thread/read` at predictable elapsed checkpoints so a missed terminal notification can be reconciled from authoritative thread state without inventing completion.
- Late provider snapshot backfill or daemon replay can recreate stale assistant `is_streaming` flags after an earlier one-shot migration. Startup migrations may re-run lightweight terminal-turn streaming cleanup, but heavy provider-daemon journal pruning belongs in non-blocking runtime maintenance after IPC readiness.
- `thread/status/changed` with `status.type = "idle"` is a reconciliation trigger, not terminal proof. Cafe must call `thread/read` and clear `activeTurnId` only when the same active turn has a terminal status in that authoritative snapshot; this avoids both stale active-turn warnings and premature closure of a live turn.
- After `turn/start`, `turn/steer`, and active-turn `thread/resume`, delayed `thread/read` polling may backfill missed Codex terminal events, but only when upstream returns a terminal turn status. Upstream documents `turn/diff/updated` as a diff snapshot, so Cafe must not treat provider-diff placeholders or completed items as authoritative turn completion. If polling still sees `inProgress`, emit `codex.turnProgress/stillInProgressAfterSnapshotPolling` diagnostics and keep the turn running until Codex emits `turn/completed`, `turn/interrupt` finishes, or `thread/read` reports a terminal status. Polling continues past the first 300-second warning so late terminal snapshots can still reconcile the session.

Active app-server sessions should still prefer upstream `account/rateLimits/read`/`updated` protocol data when available.

## Claude Integration

Cafe Code's Claude adapter uses `@anthropic-ai/claude-agent-sdk`.

Claude lifecycle facts to preserve:

- The TypeScript SDK `query()` API returns an async generator that streams `SDKMessage` values as they arrive.
- `query()` can take an `AsyncIterable<SDKUserMessage>` prompt, which Cafe Code uses as a prompt queue for long-lived sessions.
- Official Claude Agent SDK streaming input mode is the supported interactive path for queued messages: the docs describe dynamic message queueing, real-time interruption, and natural multi-turn conversations for `AsyncIterable<SDKUserMessage>` prompts, while the installed SDK types document `streamInput()` as the multi-turn input pipe. Cafe therefore advertises Claude live steer support by validating Cafe's expected active turn id locally and queueing the follow-up into the active SDK prompt stream. Claude does not expose a Codex-style expected-turn RPC, so never claim upstream has acknowledged a specific turn beyond Cafe's local active-turn binding.
- The returned `Query` supports operations such as `interrupt()`, `setModel()`, `setPermissionMode()`, and `setMaxThinkingTokens()`.
- Permission mode has two distinct paths in upstream Claude Agent SDK: initial `query()` options and `Query.setPermissionMode()` for an already-active streaming session. Cafe Code must bind the first turn's interaction mode into `query()` when starting a Claude session and must not send a redundant pre-prompt `setPermissionMode()` for default/full-access sends, because current Claude Code can reject that control request before it has a transcript message/conversation to attach it to.
- Fresh Claude sessions should let upstream `query()` allocate the session id. The official SDK docs mark `sessionId` as optional/default auto-generated and recommend capturing the durable id from `system` init or result messages before passing it back through `resume`; Cafe Code should therefore only pass `resume` when it has a durable resume cursor from a real Claude transcript. Do not generate arbitrary `sessionId` values for new long-lived AsyncIterable sessions, because current Claude Code can reject the first queued turn with "No conversation found with session ID" before the transcript exists.
- Do not pass Claude SDK `resumeSessionAt` for ordinary follow-up turns. Upstream SDK types document `resumeSessionAt` as a specific assistant-message checkpoint, not as the normal latest-leaf resume pointer; always applying it to Cafe follow-ups can make current Claude Code reject valid transcripts with "No message found with message.uuid". Only use `resumeSessionAt` for an explicit user-visible rewind/fork-to-message operation with dedicated recovery handling.
- Claude `resume` loads a local transcript from the resolved Claude home project directory for the active cwd, not from Cafe's UI state alone. Before passing a durable Cafe cursor through SDK `resume`, verify that `~/.claude/projects/<encoded-cwd>/<session-id>.jsonl` exists in the resolved Claude home or can be copied from another Claude project directory. If the transcript is missing, drop the stale cursor and start a fresh upstream Claude session rather than sending a doomed `--resume` that fails before the user's turn starts.
- A zero-turn Claude `error_during_execution` result is a pre-run failure, not a durable conversation turn. Do not replace a healthy resume session id with the failed result's new session id, do not increment the durable resume turn count for that failed turn, and repair stale Cafe cursors from local Claude transcript files when a stored assistant checkpoint identifies the real session. Recoverable provider lifecycle failures should be recorded as work-log/debug diagnostics rather than promoted to blocking toasts or thread-level fatal banners.
- Claude permissions flow through SDK permission callbacks such as `canUseTool`; respect the supplied abort signal, tool-use id, suggested permission updates, and deny/allow semantics.
- Claude Agent SDK sessions persist by default unless configured otherwise. Resume/session identifiers must be treated as durable provider state, not UI state.
- Claude can emit delayed messages or wake back up after apparent quiescence. UI and projections must tolerate late provider events and should expose diagnostics when a provider speaks after a terminal-looking state.
- Claude can briefly project `running` with `activeTurnId = null` after a completed turn while the next SDK prompt is being queued and before `turn.started` supplies Cafe's local turn id. The renderer follow-up queue must not treat that newer running/null snapshot as ready just because the latest visible turn is completed; keep queued user input waiting until Claude reports a concrete active turn id or a settled session. Otherwise the renderer can race the backend's one-active-turn invariant by dispatching `thread.turn.start` into a session that is still unsettled.
- Claude SDK `system/thinking_tokens` messages are approximate live telemetry from redacted thinking deltas, not authoritative billed token usage. Cafe should keep the raw native event for logs but must not emit an unhandled-subtype runtime warning or project it as context-window usage.
- As of Claude Code v2.1.173, upstream emits `system/task_updated` patch messages alongside `task_started`, `task_progress`, and `task_notification`. The installed SDK types may lag behind the binary, so Cafe must accept known runtime subtypes observed from the current Claude Code package even when local TypeScript declarations have not caught up. Map terminal `task_updated.patch.status` values such as `completed`, `failed`, `stopped`, `killed`, or `cancelled` into `task.completed`; map non-terminal updates into `task.progress`; keep the full upstream message only in raw native diagnostics and do not emit unhandled-subtype warnings for valid task updates.
- As of Claude Code v2.1.177, the binary contains an internal `get_context_usage` control path, but it reports unsupported unless an upstream callback is registered. Cafe must not invent a polling RPC for it. For the chat composer/context-window tooltip, treat Claude `stream_event` `message_start`/`message_delta` usage and assistant snapshot usage as the live current-window estimate, then use final `result.usage` only as accumulated `totalProcessedTokens` plus `result.modelUsage[*].contextWindow` metadata. `task_progress`, `task_updated`, and `task_notification` `usage` payloads describe background/subagent task work and belong in the work log only; never project task-only counters such as `total_tokens`, `tool_uses`, or `duration_ms` as thread context-window usage.
- Claude Code stderr lines beginning with `[ede_diagnostic]` are internal execution summaries that can appear during healthy tool-use turns. Cafe must drop them before runtime warnings, work logs, toasts, and thread error surfaces instead of treating `stop_reason` values such as `tool_use` as user-visible failures.
- Claude built-in model metadata is data-driven from `apps/server/src/provider/Layers/ClaudeModelCatalog.json`. Add new first-party Claude model slugs, display names, option descriptors, context-window selectors, and minimum Claude Code version gates there rather than hard-coding another model entry in `ClaudeProvider.ts`. Cafe decodes that catalog at startup, hides entries whose `minimumClaudeCodeVersion` is newer than the installed CLI, and shows an upgrade message for the newest hidden model. Keep `packages/contracts/src/model.ts` aliases in sync when upstream changes moving aliases such as `opus` or `fable`. As of Claude Code v2.1.170, the installed CLI exposes `claude-fable-5`, `fable`, and `fable[1m]`, plus usage-credit launch strings but no structured settings schema beyond the existing model/effort/context controls. Cafe gates `claude-fable-5` on Claude Code v2.1.170, treats it like Opus 4.8 for settings with `xhigh` as the default effort, `max` as the highest SDK effort, `ultrathink` as prompt-injected, and 200k/1M context-window choices, and appends `[1m]` through `resolveClaudeApiModelId` rather than storing bracketed upstream aliases as canonical models. As of Claude Code v2.1.154, upstream maps `opus` to `claude-opus-4-8`; Cafe's catalog exposes `claude-opus-4-8` with `xhigh` as its default effort, `max` as the highest effort, `xhigh`/`max` passed through to the CLI/SDK when selected, and 200k/1M context-window choices.
- Claude provider launches must set an explicit `CLAUDE_CONFIG_DIR` for the effective Claude home while preserving a user-configured provider-instance `CLAUDE_CONFIG_DIR`. This keeps Cafe SDK sessions aligned with verified CLI launches such as `CLAUDE_CONFIG_DIR=/Users/me/.claude claude`. If Claude emits `api_retry`/result authentication failures with HTTP 401, treat the active SDK session as stale login material: record work-log diagnostics, suppress Claude Code's synthetic assistant error text, close the stale session, and let the next turn start a fresh Claude process with current credentials.

## Observability And Debugging

- Every provider lifecycle change must leave enough debug breadcrumbs to answer: what command was sent, when it was accepted, when the provider emitted the first runtime event, when the first token/content/tool event arrived, when it completed, and what process owned it.
- The debug endpoint should include provider daemon health, provider supervisor health, process diagnostics, RPC metrics, command ledger summaries, recent failures with stack traces where available, runtime event method counts, recent lifecycle warnings, stale active turns, provider home paths, and provider process ownership.
- Settings diagnostics may show lightweight runtime-layer process, daemon, supervisor, and orchestration counters without `--cafe-debug`, but those summaries must stay bounded and sanitized. Keep raw prompts, full model output, bearer tokens, auth files, environment dumps, and deep event replay behind explicit local debugging flows.
- Add new debug fields when fixing new failure classes. Do not rely on logs alone when a compact health/debug summary would make future diagnosis faster.
- Runtime event names should be specific enough to grep, such as `codex.turnStart/accepted`, `codex.turnStart/noRuntimeEventYet`, provider stream disconnect warnings, snapshot backfill markers, and post-completion late-event diagnostics.
- Prefer structured error diagnostics with cause chains and stack traces. If an exception crosses a daemon/RPC boundary, preserve its type, message, stack when safe, and structured context.
- For performance issues, instrument before guessing. Track send-to-ACK latency, ACK-to-runtime-start latency, runtime-start-to-first-token latency, tool-call duration, database wait/busy time, projection latency, WebSocket push latency, renderer receipt timing, store/reducer time, render commit time, and token-to-screen latency.
- Debug continuation heuristics must treat the latest provider turn terminal event as the authoritative completion boundary. Older checkpoint/diff summaries for the same turn are artifact metadata, not proof that later same-turn tools or assistant messages were post-completion corruption.

## Persistence And SQLite

- Keep provider command ledgers idempotent. Mutating provider daemon RPCs must carry command IDs when replay or retry is possible.
- Avoid long SQLite transactions around provider I/O, process startup, network calls, or stream consumption.
- Do not load entire chat histories or event stores on every token/tool call. Use bounded queries, projections, cursors, and summary snapshots.
- Keep provider daemon event journals bounded in SQLite as well as in memory. The persistent `provider_daemon_events` table is a restart/replay buffer for daemon/supervisor handoff, not an analytics warehouse; retaining unbounded provider events causes post-prompt latency and stale projection pressure that the Codex CLI does not have. Large inherited journals must be pruned by non-blocking journal maintenance after IPC/TCP readiness, because one-time migration deletes can exceed the desktop provider-daemon readiness timeout and leave bootstrap without a socket. Restart replay from a detached supervisor must be idempotent on provider runtime `eventId`, not local cursor, because supervisor cursors and daemon cursors are separate ownership domains.
- SQLite `database is locked` is a real lifecycle/performance signal. Add diagnostics for the writer, SQL operation, busy timeout, and affected command path instead of only increasing timeouts.
- Migrations must be deterministic, backward-safe where possible, and covered by tests. Reconciliation migrations should explain the exact stale state they repair.

## Performance Rules

- Cafe Code must be designed as a long-running provider stream processor, not a conventional short chat UI. Multi-hour and 16+ hour turns are expected. Small per-token costs compound into user-visible lag over those runs, so every token/event hot path must account for token-to-screen time, render-loop pressure, and accumulated projection cost.
- Provider token streaming must stay streaming. It is acceptable to coalesce tiny token deltas into small bounded projection chunks after the first visible bytes, but never batch tokens behind expensive projection work, full-history reads, synchronous filesystem scans, or renderer-only timers.
- Keep hot paths allocation-conscious and query-bounded. Any per-token or per-tool-call code should be reviewed as a performance-critical path.
- Provider event ingestion, projection, WebSocket fanout, renderer state updates, markdown rendering, scroll anchoring, and debug publication must stay independently bounded. Do not introduce work that scales with full chat history, full activity history, or total turn duration on each token/tool update.
- Do not perform broad process scans, recursive filesystem traversal, or source-control operations inside message send or provider event ingestion paths unless explicitly debounced and instrumented.
- Hidden checkpoint ref pruning can run after turn completion, so it must stay bounded: delete refs in batches with `git update-ref --stdin`, validate generated checkpoint ref names before stdin use, and log only counts plus small samples on prune failure. Projection-only placeholders such as `provider-diff:*` are not git refs; filter them before retention math and deletion so they cannot poison a whole prune batch or make Cafe retain/delete the wrong hidden checkpoint turns. Never spawn one git process per historical checkpoint ref or emit thousands of refs into runtime logs.
- Checkpoint capture only stages tracked content. It must first enumerate changed tracked paths and then update Cafe's temporary checkpoint index with path-limited `git add -u -- <changed-paths>` batches; do not use whole-worktree `git add -u -- .` in normal checkpoint capture, because large repositories can spend tens of seconds of CPU on every provider turn. Untracked-only repositories should produce an empty hidden checkpoint instead of a fatal checkpoint loop. Do not broaden checkpoint capture to `git add .`, because untracked files can include secrets or user scratch data.
- When comparing performance to Codex CLI or Claude CLI, identify the comparable phase first: process startup, session resume, prompt submission ACK, provider runtime start, model first token, tool execution, or UI projection.

## Frontend Lifecycle Rules

- The renderer should display backend/provider truth and should not synthesize terminal, running, or active-turn state that can conflict with orchestration projections.
- Scroll-follow behavior should be tolerant of small gaps from the bottom and must not jump to the top when steer messages, late messages, or terminal markers arrive.
- Message timelines must handle late provider events after completion without duplicating terminal banners or losing streamed content.
- Per-thread detail subscriptions must apply snapshots and events monotonically by orchestration sequence. Events at or below the detail snapshot sequence are stale for that subscription and must not regress focused-thread session or turn state.
- Codex steer UI state should mirror upstream CLI semantics: a `turn/steer` ACK is not enough to clear a pending steer marker. Keep the renderer in a visible "steering" state until Codex emits the provider-side user-message/processing-start signal, a terminal turn, or a documented recovery/failure event.
- Shell and per-thread detail WebSocket subscriptions are critical recovery streams. They must retry recoverable stream/protocol failures, force a fresh snapshot after reconnect, and never leave the renderer pinned to stale active-turn state while the backend projection/database has already advanced. Heartbeat timeouts should recycle the WebSocket session so subscriptions reattach instead of waiting on a half-dead socket.
- Recoverable provider, transport, and auto-retry diagnostics should not interrupt the user with toasts. Surface them in the work log, thread banners, connection diagnostics, or debug output unless immediate user action is required.
- UI disconnects, focus changes, reloads, or renderer stalls must be projection-catch-up events, never provider-session truth. The renderer should reconnect from a cursor or bounded snapshot, preserve queued follow-up/steer intent, and must not restart, interrupt, or orphan provider work merely because the local render loop lagged.
- Work-log rendering keeps the latest/current turn inline from the thread detail snapshot. Older turns render as collapsed inline `Work log` rows between that turn's user/assistant messages and hydrate their activity pages on demand through `orchestration.getThreadTurnActivityPage`; do not expand the thread detail snapshot activity cap to show historical logs.
- Historical work-log count/page queries must count only activities that render in the work log. Exclude context-window bookkeeping, checkpoint-captured markers, `task.started`, ordinary `tool.started`, retryable steer-delivery placeholders, and ExitPlanMode boundary tool rows from the DB-backed historical page path. If the on-demand count proves a placeholder historical row has no displayable entries, the renderer should remove that row instead of expanding to an empty panel.
- Retryable Codex `activeTurnNotSteerable` follow-ups for `review` or `compact` are automatic steer retries, not ordinary queued messages. The renderer should label them as waiting to steer, hide manual Send actions that could duplicate the user bubble, retry compact-blocked steers only after the active context compaction item completes, and retry review-blocked follow-ups only when the active turn is ready for the next message.
- Renderer session reducers must treat same-turn `running` snapshots after terminal `ready`/`error`/`interrupted` state as stale provider replay. Preserve the terminal latest turn, active-turn clearing, and newest lifecycle timestamp; otherwise debug/work-state can falsely report post-completion messages or a stale steerable active turn even though the backend projection is correct.
- Keep renderer token streaming on a tight hot path. Thread-detail stream events may be frame-batched before they enter Zustand, but sequence guards must still preserve monotonic projection order. Only append-like streaming deltas may be coalesced together; terminal assistant completion events must remain separate because non-streaming message events use authoritative replace/preserve semantics. Existing message deltas should update normalized message maps directly instead of materializing the full `Thread`, scanning the entire 2,000-message window, and rebuilding every slice per token.
- Desktop debug snapshots are diagnostic output, not provider or UI truth. When `--cafe-debug` is enabled, renderer debug publication must be throttled/heartbeat-driven so large debug JSON snapshots do not compete with token streaming, timeline rendering, or follow-up/steer dispatch.
- Assistant output may arrive as sparse or final large deltas even when the provider is healthy. The backend should keep streaming chunks small and bounded; the renderer may smooth display of an already-received large assistant append for readability, but it must not synthesize provider lifecycle state or hide backend/provider timing diagnostics.
- Live steer UI must distinguish queued follow-ups from accepted in-flight steers. Once Cafe dispatches a provider-supported steer into an active turn, show a non-cancelable steering state until provider output, terminal turn state, or an explicit steer failure/requeue proves the handoff is resolved; otherwise users cannot tell whether Codex/Claude received the follow-up or the renderer lost it.
- User-facing follow-up controls must not tell the user that a provider cannot steer, cannot accept input, or can only queue. Accept the user's intent, then route it through live steer, queued follow-up, interrupt-and-resume, app-server restart, or another documented recovery path automatically. Provider limitations, active-turn mismatch details, review/compact rejections, and retry metadata belong in debug/work-log diagnostics, not as blocking UI copy.
- Toasts should be reserved for action-required or non-recoverable conditions. Auto-retryable provider issues, delayed steers, queued follow-ups, compaction waits, and recoverable transport/provider races should update the work log, queue UI, or debug diagnostics instead of popping transient toasts.
